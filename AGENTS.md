# SES AI Assistant Core Library — 開発ガイドライン

本ドキュメントは、本プロジェクトのメンテナンスをするすべての開発者向けの必読ガイドです。

## セッション開始時の読み取り順（必須）

1. このファイル（AGENTS.md）全体を読む
2. `TODO.md` を読み、現在進行中のタスクを把握する
3. 必要に応じて、影響するファイルのみを開く

---

## 1. プロジェクト概要

本プロジェクトは、**SES AI ASSISTANT システム全体の共通基盤ライブラリ**です。RDB/NoSQL エンティティ定義、ドメインロジックの最小単位（Value Object/Unit）、および外部サービス（AWS, LLM, Slack等）との通信を統一的に提供します。

このライブラリの品質と設計の整合性は、**他の8つのリポジトリ（Lambda、Backend、Batch）の土台**となるため、最高の責任を持って開発してください。

- **言語**: Java 17
- **フレームワーク**: なし（ライブラリ）
- **ビルドツール**: Maven
- **IDE**: Eclipse
- **テスト**: ユニットテストのみ（IT1 なし）
- **CI/CD**: GitHub Actions

## 2. 設計思想・アーキテクチャ

既存の設計思想を厳格に遵守し、システム全体の整合性を保つことを最優先とします。

### 2.1 データベース・エンティティ（`database` パッケージ）

**Entity/Lot パターン**:
- 単一レコードを表す `SES_AI_...` クラスと、そのリストおよび操作を管理する `SES_AI_...Lot` クラスが対になっています
- 新しいテーブルを追加する場合は、`EntityBase` または `SES_AI_T_EntityBase` を継承し、必ず対応する `Lot` クラスを作成してください

**DynamoDB対応**:
- `database.base.DynamoDB` を基底とした抽象化を利用し、RDB と一貫性のある操作感を目指しています

### 2.2 ドメイン・ユニット（`unit` パッケージ）

**Value Object の徹底**:
- `Money`, `Currency`, `OriginalDateTime` など、値とロジックをカプセル化したクラス（Domain Primitive）を積極的に利用してください
- プリミティブ型（String, int等）を直接扱うのではなく、可能な限り `unit` パッケージのクラスへ変換して扱います

### 2.3 外部API連携（`api` パッケージ）

**抽象化レイヤー**:
- LLM（OpenAI, Gemini）などは `Transformer` や `GptModel` といったインターフェース/抽象クラスを介して利用します
- 特定のベンダーに依存した実装を直接ビジネスロジックに書かず、`api` パッケージ内のプロキシクラスを通してください

## 3. 実装ルール

### 3.1 基本的なコーディング規約

- **命名規則**: クラス名は既存の `SES_AI_` プレフィックスや、用途に応じた接尾辞（Lot, Tests）を遵守
- **エラーハンドリング**: `logback.xml` の設定に基づき、適切なログレベルで記録。例外を握り潰さず、上位レイヤーに適切に伝播させるか、ドメイン例外にラップ
- **型安全**: ジェネリクスを適切に使用し、ランタイムエラーを最小限に抑える

### 3.2 テスト1対1ルール（必須）

- **テストクラス**: 1 つのクラスに対して、必ず 1 つのテストクラスを `src/test/java` に作成してください
- **複数ファイル禁止**: テストケースを複数のファイルに分離させることは **禁止** です
- **カバレッジ 100%**: すべてのテストが PASS し、プロジェクトが求める**カバレッジ 100%** を満たすことが必須

## 4. テスト方針

### 4.1 ユニットテスト

**回帰テストの重視**:
- 既存の `src/test/java` 内のテストケース（`GeminiTests`, `SES_AI_...Tests` 等）を、修正後に必ず実行し、デグレードがないことを確認してください

**新規実装時のテスト**:
- 新機能やバグ修正を行う際は、必ず対応するテストクラスを `src/test/java` に作成または追加してください

## 5. ビルドとテスト

```bash
# コンパイル
mvn clean compile

# ユニットテスト実行（カバレッジ確認）
mvn test

# パッケージ（JAR 生成）
mvn clean package

# 静的コード解析
mvn rewrite:run          # 未使用インポート削除・コードスタイル修正
mvn spotless:apply       # コードフォーマット
mvn checkstyle:check     # チェックスタイル確認
mvn pmd:pmd              # 品質解析
mvn pmd:cpd              # 重複コード検査
mvn spotbugs:check       # バグ検出
```

## 6. メンテナンス時の注意点

- **依存関係**: `pom.xml` を確認し、ライブラリの追加が必要な場合は既存のバージョンとの競合に注意
- **共通性**: 特定のアプリケーションに特化したロジックはここには含めず、常に「複数システムで再利用可能か」を基準に判断
- **波及範囲**: このライブラリの変更は、他の8つのリポジトリに波及する可能性があります。変更後は、該当するリポジトリの README を確認し、矛盾がないか検証してください

## 7. AI Agent 運用ルール（必須）

Gemini CLI や Claude Code がこのプロジェクトのコードを生成・修正する際は、以下のワークフローを厳守してください。

### 7.1 作業開始時の「儀式」

1. **指示の確認と抽出**: `docs/INSTRUCTION.md` を読み込み、**最も新しく追加された未完了のタスク**を特定
2. **TODO.md への転記**: 特定したタスクを `TODO.md` に具体的なステップ（調査・実装・検証）として書き出す
3. **シングルタスクの徹底**: 同時に複数の大きなタスクを進めない
4. **完了後のクリーンアップ**: タスクが完了したら、実行した指示内容を `docs/ARCHIVE_INSTRUCTIONS.md` に追記し、`docs/INSTRUCTION.md` をテンプレート状態に戻す
5. **コンテキスト同期**: 開発を再開する際は必ず `read_file("TODO.md")` を行う
6. **設計書最新化**: 開発が完了したら必ず `docs/` 配下の設計書のうち、更新が必要なもののみを更新

### 7.2 思考の連鎖 (Chain of Thought)

- **事前調査**: コードを書く前に、必ず以下の内容を調査・報告してください
  - `unit/` や `api/` パッケージ内に既存クラスが再利用可能か
  - 修正対象のファイルと、その影響範囲（特に依存する Lambda/Backend）
- **実装方針の提示**: `replace` や `write_file` を呼ぶ前に、実装のステップを箇条書きで提示

### 7.3 実装の粒度と共通部品の優先

- **DRY原則**: 新しいロジックを作る際は、まず `unit/` 内に既存のドメイン型がないか、`util/` に共通メソッドがないか確認してください。あればそれを優先して使用し、なければ共通部品としての切り出しを検討
- **実装密度**: 1クラスの肥大化を避け、複雑になる場合は `Lot` クラスや別のドメインサービスに委譲
- **差分更新の徹底**: ファイル全体を書き換えるのではなく、`replace` ツールで修正が必要なメソッド単位で変更

### 7.4 自律的検証

- **自動テスト**: 修正後は必ず `mvn test` で成功を確認。カバレッジは 100% を常に担保
  - **テスト1対1ルール**: 1 つのクラスに対して、必ず 1 つのテストクラスであること
  - **複数ファイル禁止**: テストケースを複数のテストファイルに分離させないこと
  - `mvn clean install` でビルドが通ることを確認
  - `mvn test` でテストが通りカバレッジを確認

- **リファクタリング**: テスト通過後、命名規則やコードの綺麗さをセルフチェック

- **静的コードチェック**: `mvn rewrite:run`, `mvn spotless:apply`, `mvn checkstyle:check` 等を実行

### 7.5 波及範囲の確認

- **他リポジトリへの影響**: このライブラリの変更が依存する8つのリポジトリに影響する可能性があります
- **検証方法**: 変更後、該当するリポジトリ（Lambda, Backend, Batch）の README や AGENTS.md を確認し、矛盾がないか検証
- **必要に応じて**: 依存リポジトリの README も更新してください
